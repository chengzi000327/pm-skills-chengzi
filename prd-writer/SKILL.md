# PRD 写作 Skill

## 核心原则

**说人话**。PRD 不是八股文，不要堆砌"以用户为中心""赋能业务增长"这类空话。每一句话都要有信息量。

**写给开发看**。PRD 的核心读者是工程师和测试。逻辑清楚、边界写明、字段定义清晰，比排版好看重要一百倍。

**细节决定成败**。模糊的需求是 bug 的温床。字段类型、校验规则、异常处理、状态流转——这些都要写清楚，不能留白让研发自行发挥。

---

## 第一步：识别功能类型

收到需求后，**先判断功能类型**，再选对应的章节模板。判断依据：

| 类型 | 判断信号 | 典型例子 |
|------|---------|---------|
| **Agent / AI 类** | 涉及大模型调用、意图识别、多轮对话、工具链、自动化执行 | AI 记录助手、智能客服、对话式功能 |
| **普通功能类** | 有明确的业务逻辑、数据库读写、接口交互，但不涉及 LLM | 订单系统、消息推送、权限管理 |
| **纯交互 / 前端类** | 重点是页面设计、交互流程、视觉反馈，后端逻辑简单或已有 | 新版 UI 改版、引导流程、弹窗组件 |

**如果功能混合**（比如有 AI 能力又有复杂后端逻辑），以最复杂的部分为主类型，其余作为子章节处理。

**如果判断不确定**，直接用「普通功能类」模板，它是最通用的底座。

### 3. 功能概述

四个维度，每条 1-3 句话，不要水：

- **背景**：为什么做？现在有什么问题？能用数据就用数据，比如"当前成功率只有 65%，主要原因是..."
- **目标用户**：谁在什么场景下用？
- **功能介绍**：能做什么？一句话概括。
- **目标收益**：做完之后指标能达到多少？要有具体数字。

### ▌类型 A：Agent / AI 类

#### 4. 流程定义

**4.1 业务流程**

用 show_widget 画 SVG 流程图，蓝色=工程节点，绿色=LLM节点，红色=异常路径，分支箭头标注条件文字。流程图下方紧跟节点说明表，两者互补：

| Node | 类型 | 流转（Edge） | 备注 |
|------|------|-------------|------|
| 等待用户输入 | 工程 | → 记忆与State获取 | 新任务或补充输入 |
| 意图识别 | LLM | 成功 → 执行；拒绝 → 结束 | 单轮识别，失败可重试一次 |
| 技能执行 | 工程+LLM | → END | 具体技能抽象汇总 |
| END | 工程 | — | 更新State，生成结构化输出 |

**4.2 Agent 关键节点说明**

每个 LLM 节点单独说明输入/输出/异常处理。

#### 5. 功能点详细说明

必须包含的子章节：

- **5.1 Agent State 设计**：State 是流程流转的核心，所有节点都读写它。

  | 分类 | 字段 | 类型 | 必填 | 说明 |
  |------|------|------|------|------|
  | 基础 | trace_id | string | 是 | 链路追踪ID |
  | 基础 | user_id | string | 是 | 用户ID |
  | 回合状态 | intent | string | 是 | 当前意图枚举值 |
  | 回合状态 | need_clarification | boolean | 是 | 是否需要用户补充 |
  | 执行控制 | max_transitions_per_turn | integer | 是 | 最大转换次数，防死循环，默认10 |

- **5.2 意图识别**：输入构成、输出格式、解析方式、各意图的后续流转。

- **5.3 技能定义**：每个技能的节点流转表、输入输出字段、校验规则。

- **5.4 异常处理**：系统级异常的兜底逻辑（不是业务异常，是运行崩了怎么办）。

---

**模板（每个 Prompt 都要写齐以下各项）：**

```
### 6.X [Prompt名称]

**用途**：一句话说清楚这个 Prompt 做什么、在哪个节点调用。

**System Prompt 结构**：

System Prompt 按以下顺序组织，缺哪块就省哪块，顺序不能乱：

1. 角色定义
   你是一个 xxx 助手，擅长 xxx。
   → 一句话，不要堆砌形容词。

2. 任务目标
   根据用户输入的 xxx，完成 xxx 任务，输出 xxx。
   → 说清楚输入是什么、要做什么、输出是什么。

3. 判断逻辑 / 优先级（CoT：让模型显式推理，不要直接跳结论）
   复杂判断场景下，要求模型先输出推理过程，再给结论。
   写法：
     - "请先分析用户输入属于哪种情况，再输出意图"
     - "先判断图片是否包含有效数据，再决定是否继续识别"
     - "按以下优先级判断：1. xxx 2. xxx 3. xxx，逐条检查后输出结论"

   注意：如果输出格式中有推理行（如 r|原因 这种），就是把 CoT 写进了输出格式里，
   不需要在 System 里再重复要求"先思考"。

4. Few-shot 示例（适用场景：输出格式复杂、边界 case 多、模型容易跑偏时）
   在 System Prompt 末尾或单独的 user/assistant message 对里放示例。
   
   原则：
   - 至少给 1 个正常示例 + 1 个边界/异常示例
   - 示例要覆盖最容易出错的 case，不要只给最简单的情况
   - 示例格式必须和真实输出格式完全一致，不能有任何差异
   
   放在 System 末尾的写法：
     ## 示例
     输入：今天早上吃了两个鸡蛋
     输出：
     r|文字输入，包含食物描述
     i|diet

     输入：帮我查一下明天天气
     输出：
     r|与记录无关，属于查询类请求
     i|reject

   放在独立 message 对里的写法（适合示例较长时）：
     { "role": "user", "content": "示例输入 A" }
     { "role": "assistant", "content": "示例输出 A" }
     { "role": "user", "content": "示例输入 B" }
     { "role": "assistant", "content": "示例输出 B" }
     → 放在 system 之后、真实历史记忆之前

5. 输出格式约束
   在 System Prompt 里明确写输出格式要求，不要只在 PRD 文档里说。
   写法：
     ## 输出格式
     - 纯字符串，两行，\n 分隔，不要有多余内容
     - 第一行：r|#推理原因#
     - 第二行：i|#意图枚举值#（只能是 diet / sport / weight / reject 之一）
     - 不要输出任何解释、markdown、代码块

6. 限制 & 兜底
   明确告诉模型不能做什么，以及遇到模糊情况的默认行为。
   写法：
     ## 限制
     - 若无法确定意图，输出 i|reject，不要猜测
     - 不要编造数据，识别不到的字段输出 NA
     - 不得输出 markdown 格式，不得用代码块包裹输出

**输入 Message 构成**：

| role | 内容 | 说明 |
|------|------|------|
| system | [Prompt名称] 完整提示词 | 包含上述1-6项 |
| user/assistant | Few-shot 示例对（若示例较长） | 放在 system 后、记忆前 |
| user | 参考数据（如近3日记录） | 仅在有参考数据时传入 |
| user/assistant | 历史对话记忆，至多20轮 | ChatML格式，按记忆规范读取 |
| user | 上一轮待澄清问题（若有） | 从 pending_validations 中提取 |
| user | 用户当前输入（文本+图片） | 本轮实际输入内容 |

**输出格式**（必须给真实示例，不能只写"输出JSON"）：

格式说明：
- 纯字符串，xxx 行，\n 分隔
- 第一行：xxx|#内容#
- 第二行：xxx|#内容#

示例输出：
r|用户本轮输入是对上轮饮食记录的补充
i|diet

**解析方式**（有特殊解析逻辑的必须写出来）：

def parse_xxx(llm_output: str):
    match = re.match(r'r\|(.+)\ni\|(\w+)', llm_output.strip())
    if not match:
        raise Exception('解析失败')  # 视为 ERROR，重试一次
    reason, intent = match.groups()
    return reason, intent

**边界 & 限制**：
- 识别失败时怎么处理（重试/直接报错/兜底值）
- 输出为空或格式异常时的处理
- 特殊输入的处理规则（如无图片、多图、历史为空等）
```

---

**什么时候用 CoT，什么时候用 Few-shot**：

| 场景 | 用哪个 | 原因 |
|------|--------|------|
| 多条件判断，容易跳步 | CoT | 让模型逐条推理，减少错判 |
| 输出格式复杂，模型容易格式跑偏 | Few-shot | 用示例锚定格式 |
| 边界 case 多，文字描述说不清楚 | Few-shot | 示例比规则更直接 |
| 输出已包含推理行（如 r\|原因） | 不用额外加 CoT | 输出格式本身就是 CoT |
| 简单分类，case 清晰 | 都不需要 | 加了反而增加 token 消耗 |

---

**三类常见 Prompt 的 Message 构成差异**：

| 类型 | system | 参考数据 | 记忆 | 待澄清 | 用户输入 |
|------|--------|---------|------|--------|---------|
| 意图识别 | ✅ | ✗ | ✅ 20轮 | ✅ 上轮未完成项 | ✅ 文本+图片 |
| 数据识别 | ✅ | ✅ 近期记录 | ✅ 20轮 | ✅ 上轮未完成项 | ✅ 文本+图片 |
| 字段补充 | ✅ | ✗ | ✅ 20轮 | ✗ | ✅ 待补充字段列表 |
| 去重/合并 | ✅ | ✅ 已保存记录 | ✅ 20轮 | ✗ | ✅ 本轮识别结果 |

**输出格式的几种常见形式**（选一种，在 Prompt 里定义清楚）：

```
# 形式1：竖线分隔的多行文本（适合简单结构）
d|2025-10-23
m|午餐
f|n=鸡蛋|q=2|u=个|c=140|ch=2|p=12|fat=10

# 形式2：带前缀的两行（适合意图识别类）
r|用户本轮输入是对上轮饮食记录的补充
i|diet

# 形式3：索引保留列表（适合合并/去重类）
a|分析说明文字
k|0|2|3

# 形式4：JSON（适合字段多、结构复杂的场景，但要明确禁止 markdown 代码块包裹）
{"field": "weight", "value": "62", "reason": "out_of_range"}
```

**关于记忆的处理**（所有 Prompt 通用规范）：
- 历史消息按 ChatML 格式处理，至多取最近 20 轮

- 若历史为空，直接省略 memory message，不传空数组

## 写作风格指南

**说人话的核心是降低读者的理解成本**，不是口语化。以下六条是具体操作规范：

---

### 1. 动宾结构，少用长句

描述功能时，用"操作 → 触发 → 结果"的链式写法，不写长难句。

✅ `点击【发送】按钮 → 校验表单 → 成功则提交，失败则在对应字段下方展示错误提示`
❌ `用户在完成表单填写后可以通过点击发送按钮来提交数据，系统会对数据进行校验`

---

### 2. 少用代词，直接指名道姓

文档里少用"它"、"该功能"、"上述操作"，防止歧义。用【功能名】、「按钮名」直接指代。

✅ `【实名认证】流程完成后，「提交」按钮变为可点击状态`
❌ `该流程完成后，它会变成可点击状态`

---

### 3. 分点陈述，能列表不写段落

能用列表解决的，绝不写一段长文字。

✅
```
触发条件：
- 用户未登录时访问该页面
- token 过期
- 主动点击「退出登录」
```
❌ `当用户未登录或 token 过期时，或者主动点击退出时，系统会跳转到登录页。`

---

### 4. 用户场景驱动，说清楚"为什么"

开发最烦的是"做一个搜索框"，却不知道为什么要搜、搜索逻辑是什么。

✅ `当用户只记得模糊关键词时，搜索需支持模糊匹配，并优先展示用户最近购买过的类目。举例：输入"苹果"，搜索建议首位应出现其购买频率最高的"红富士"。`
❌ `用户可以在首页点击搜索框进行商品搜索。`（废话，看原型图就知道）

**复杂逻辑必须配具体 Example**，不要只写规则不给案例。

---

### 5. 描述逻辑而非 UI，写开发的盲区

PRD 不是像素级翻译机，UI 细节是设计稿的事。要写的是：**状态切换条件、数据来源、计算公式、异常反馈**。

✅ `「提交」按钮在未勾选《用户协议》时为禁用态；勾选后变为激活态；提交中变为 loading 态且不可重复点击。`
❌ `按钮是红色的，圆角 4px。`

多条件判断用**判定表**，不要用文字描述 if-else：

| 已登录 | 已实名 | 余额充足 | 结果 |
|--------|--------|---------|------|
| ✅ | ✅ | ✅ | 正常下单 |
| ✅ | ✅ | ❌ | 提示充值 |
| ✅ | ❌ | - | 跳转实名认证 |
| ❌ | - | - | 跳转登录 |

---

### 6. 消除隐性假设，把"默认"写出来

你脑子里"理所当然"的事，开发可能完全不知道。写之前主动检查：

- **默认有网吗？** → 无网络时显示什么？
- **默认已登录吗？** → 游客状态看什么、能做什么？
- **默认数据即时加载吗？** → 加载中是 loading 转圈还是骨架屏？
- **返回键行为** → 回到上一级还是回到首页？
- **数据为空时** → 空态文案是什么？有无引导操作？
- **操作不可逆时** → 有没有二次确认弹窗？

---

### 7. 文末加"备注 & 常见问题"

把评审阶段被研发、测试挑战过的问题直接写进文档，格式：

```
**Q：为什么不能跳过这一步？**
A：涉及合规审计，必须保留记录，不可省略。

**Q：用户拒绝授权位置信息后还能用吗？**
A：可以用，但搜索结果不做地域排序，默认按热度排序。
```

### 关于模糊处的处理

遇到逻辑不清楚的地方：
- **能合理推断的**：直接写，旁边标 `[TODO: 待确认]`
- **真的不清楚的**：用 `[待补充: 需确认 xxx]` 占位，文末统一汇总成待确认清单

## 流程图绘制规范

**所有流程定义章节都必须画流程图**，用 show_widget 渲染 SVG，不要只写文字或节点表。流程图 + 节点说明表并列输出，互补而非重复。

### SVG 渲染规则（严格遵守）

**viewBox 固定宽度 680**，高度根据内容自适应，最后一个元素底部 + 40px。

**箭头 marker 每张图都要加**：
```svg
<defs>
  <marker id="arrow" viewBox="0 0 10 10" refX="8" refY="5" markerWidth="6" markerHeight="6" orient="auto-start-reverse">
    <path d="M2 1L8 5L2 9" fill="none" stroke="context-stroke" stroke-width="1.5" stroke-linecap="round" stroke-linejoin="round"/>
  </marker>
</defs>
```

**字体只用两种 class**：`class="th"` 14px 加粗标题，`class="ts"` 12px 次要文字。不要写裸 `<text>` 不带 class。

**文字必须垂直居中**：`<text>` 用 `dominant-baseline="central"`，y 值设为矩形中心点。

**连线不得穿过其他节点**：需绕行时用 L 形 `<path d="M x1 y1 L x1 ym L x2 ym L x2 y2"/>`。

**深色模式强制支持**：所有颜色用 CSS color ramp 类，不要硬编码 hex。

### 节点类型与配色

| 节点类型 | class | 形状 | 说明 |
|---------|-------|------|------|
| 开始/结束 | `c-gray` | `rx=20` 圆角矩形 | 流程起止 |
| 工程节点 | `c-blue` | `rx=8` 矩形 | 代码/工程逻辑 |
| LLM 节点 | `c-teal` | `rx=8` 矩形 | 大模型调用 |
| 判断节点 | `c-amber` | 菱形（`<polygon>`）| 条件分支 |
| 异常/拒绝 | `c-red` | `rx=8` 矩形 | 错误/拒绝路径 |

**连线颜色**：`stroke="var(--color-border-secondary)"` + `marker-end="url(#arrow)"`，分支旁加 12px `ts` 标注条件。

### 不同类型的侧重

- **Agent 类**：节点颜色区分工程 vs LLM，画清分支条件，异常兜底路径用红色节点
- **普通功能类**：画清触发入口、主流程、异常/边界路径分叉
- **纯交互类**：画清用户操作路径、状态切换条件、错误恢复路径

### 节点数量与拆图

- ≤ 8 个节点：单张图，垂直布局
- 9-15 个节点：单张图，考虑左右分支布局
- > 15 个节点：拆成主流程图 + 子流程图，分开画，标注"详见子流程"

## 工作流程

1. **判断类型**：根据用户描述判断是 Agent类 / 普通功能类 / 纯交互类，不确定就默认普通功能类
2. **收集关键信息**：最多问 3 个最关键的问题，能推断的不问
3. **直接起草**：流程定义章节先画流程图（SVG），再输出节点说明表；其他章节写不清楚的用 `[待补充: ...]` 占位
4. **文末汇总**：把所有 `[待补充]` 整理成待确认清单

---

## 输出格式要求

- 直接输出 Markdown，开头不要废话
- 流程图用 SVG 通过 show_widget 渲染，紧接着输出对应的节点说明表
- 表格用 Markdown 表格，代码用代码块
- 章节最深到 `###`，不用 `####`
- 篇幅够用就行，不堆字数
