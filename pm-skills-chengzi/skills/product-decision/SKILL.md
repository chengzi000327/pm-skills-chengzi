# 产品决策 Skill（方向验证 + 立项决策）

## 概述

这个 skill 覆盖产品决策链的两个关键阶段：

```
【阶段一】方向验证  →  【阶段二】立项决策
"这个方向想清楚了吗？"    "这个产品该不该做？"
场景拆解/张力识别         AREA五层分析
输出：方向是否值得深入     输出：推进/谨慎/暂停
```

**核心原则**：
- 出色的决策源于严谨的决策过程，而非经验或直觉
- 研究是决策的基本前提，调研先行
- 重大失误来自分析失误，而非数据收集失误
- 好问题比好流程更重要
- 洞见 = 信息收集 + 大脑深度思考

---

## 第一步：判断从哪里开始

收到请求后，先判断用户所处阶段，然后告知用户将走哪个路径：

| 用户描述特征 | 走哪个阶段 |
|------------|---------|
| 方向模糊、概念阶段、"我想做 AI + X" | 从【阶段一：方向验证】开始 |
| 方向明确、有具体产品或功能描述 | 跳到【阶段二：立项决策】 |
| 两者都有或不确定 | 完整走两个阶段 |

如果信息严重不足，先输出**引导问题**（见文末）。

---

## 阶段一：方向验证

**目标**：判断"这个方向想清楚了吗"，在投入大量资源前先验证方向是否成立。

### Step 1｜场景拆解
将模糊的方向拆解为 3-5 个具体使用场景：

> 格式：[谁] 在 [什么情况下] 需要 [做什么] → 目前怎么解决 → 痛点程度（高/中/低）

逐一列出，不要合并，不同场景可能指向完全不同的产品形态。

### Step 2｜张力识别
找出方向中存在的内在矛盾或取舍困境：

常见张力类型：
- **用户张力**：不同用户群需求互相冲突
- **体验张力**：易用性 vs 专业深度
- **商业张力**：用户价值 vs 付费意愿
- **时机张力**：AI 能力现状 vs 用户预期
- **竞争张力**：差异化空间 vs 已有玩家

> 每个张力写出：[矛盾描述] → [如果选A意味着什么] → [如果选B意味着什么]

### Step 3｜方向筛选
基于场景和张力，筛选出 1-2 个值得深入的子方向：

筛选标准：
- 痛点强度足够高（用户愿意主动寻找解决方案）
- AI 在此有真实优势（非AI方案明显不够好）
- 市场存在空间（竞品有明显缺陷或未覆盖的细分）
- 我们有相对优势（数据、场景、用户关系）

### Step 4｜AREA 检验（方向层）
对筛选出的方向快速过一遍 AREA：

- **A（绝对）**：有没有第一手数据支撑这个方向的需求存在？
- **R（相对）**：行业里谁在做类似的事？我们的位置在哪？
- **E（探索）**：访谈过目标用户吗？他们的反馈是什么？
- **E（利用）**：核心假设是什么？最容易被证伪的假设是哪条？
- **A（分析）**：综合来看，这个方向值得继续深入吗？

### 阶段一输出
```
【方向验证结论】
核心场景：[1-2个最值得做的场景]
最大张力：[需要做出的关键取舍]
方向判断：
  ✅ 方向成立，建议进入立项决策
  ⚠️ 方向部分成立，需先解决[X]再推进
  ❌ 方向有根本性问题，建议重新思考，原因：[Y]
差几步：[如果选⚠️，列出需要补充的具体工作]
```

## 阶段二：立项决策（AREA 五层分析）

**目标**：对具体产品/功能做出"推进/谨慎/暂停"的系统性判断。

---

### 第一层：A — 绝对层（Absolute）
**目标：第一手客观信息，排除权威偏见和投射偏见**

**1. 关键概念澄清（1-3个）**
> 回答"我们究竟在解决什么问题"。聚焦、可量化，避免大而全。

**2. 数据审视**
- 数据是第一手（访谈/埋点）还是二手转述？
- 数据规模是否支撑判断？
- 有无背景参照（同比/行业基准）？孤立数据无意义。

**3. 问题陈述**
> "[目标用户]在[场景]中面临[痛点]，目前解决方案是[现状]，核心缺口是[X]，规模估计[量级]。"

**⚠️ 偏见预警**：投射偏见（以为用户和自己一样）、权威偏见（大厂做了就该做）、叙述偏见（被故事而非数据打动）

**→ 阶段论点**：问题是否真实存在？信息完整度？下一步补充什么？

---

### 第二层：R — 相对层（Relative）
**目标：行业视角，补全感知位置**

**行业地图（四维）**

| 维度 | 竞品/行业现状 | 与我们的差异 |
|------|------------|------------|
| 提供什么 | | |
| 什么时候（时机） | | |
| 在哪里（市场） | | |
| 多少钱（定价） | | |

**差异化判断**：竞品缺陷在哪？我们的护城河是什么？市场是否已被锁定？

**时机判断**：AI 能力是否已达门槛？监管风险是否可控？

**⚠️ 偏见预警**：社会认同偏见（跟风竞品）、相对性偏见（过度依赖比较）

**→ 阶段论点**：市场位置如何？差异化是否成立？时机是否合适？

---

### 第三层：E — 探索层（Exploration）
**目标：用户真实视角，拓展信息宽度**

**采访对象**：高价值早期用户画像是什么？是否已有 ≥5 个深度访谈？

**问题设计**

| 类型 | 示例 |
|------|------|
| 行为（现在怎么做） | "最近一次遇到这个问题，你是怎么处理的？" |
| 观点（怎么看） | "现有方案最让你不满意的地方是什么？" |
| 感受（情绪/动机） | "问题发生时，你的第一反应是什么？" |
| 知识（了解信息） | "你知道还有哪些解决方案吗？" |

**采访质量判断**：用户是主动意愿还是礼貌认可？出现了哪些出乎意料的观点？（分歧往往是关键所在）

原则：多听少说，不打断停顿，挖掘视角而非灌输想法。

**⚠️ 偏见预警**：证实偏见（只访谈支持自己的用户）、突出经验偏见（高估近期案例）

**→ 阶段论点**：用户真实意愿如何？有哪些意外发现？

---

### 第四层：E — 利用层（Exploitation）
**目标：向内审视，挑战自己的假设，增加信息深度**

**1. 核心假设列举**（至少 5 条）

**2. 竞争性替代假设**（最关键步骤）
> 关注消极因素比积极因素更重要。

| 核心假设 | 对立假设 | 支撑对立假设的证据 |
|---------|---------|----------------|
| | | |

**3. 正反分析**

| 做的理由 | 不做的理由 |
|---------|---------|
| | |

**4. 逆向思考**
> "如果 18 个月后彻底失败，最可能的原因是什么？"
列举 3-5 个失败路径 + 概率估计。

**5. 三的原则**：关键结论是否来自 ≥3 个独立信息源？

**⚠️ 偏见预警**：乐观偏见（99% 把握实际可能只有 80%）、规划谬误（低估时间成本）、损失反感（已投入资源不愿放弃）

**→ 阶段论点**：核心假设是否经得起挑战？最大不确定性在哪里？

---

### 第五层：A — 分析层（Analysis）
**目标：整合所有视角，输出最终判断**

**情景分析**（管理不确定性，非预测）

- **积极情景（18个月后）**：决策顺利的关键前提是什么？
- **消极情景（18个月后）**：失败的主要风险路径是什么？
- 如果成功，必须提前做对哪一件事？
- 这个决策 6 个月后还会成立吗？什么变化会让它失效？

**综合评分**

| 维度 | 权重 | 得分(0-10) | 说明 |
|------|------|-----------|------|
| 问题真实性与规模 | 20% | | |
| AI 必要性 | 15% | | |
| 差异化优势 | 15% | | |
| 用户访谈质量 | 15% | | |
| 技术可行性 | 10% | | |
| 商业模式清晰度 | 10% | | |
| 数据合规/伦理风险 | 10% | | |
| 团队能力匹配 | 5% | | |
| **加权总分** | 100% | | |

≥75 建议推进；55-74 谨慎推进；<55 建议暂停。

**最终判断**（三选一）

- ✅ **建议推进**：关键假设已通过多层验证，可进入 MVP 或试点
- ⚠️ **谨慎推进**：需先补充[X]，再做最终决策
- ❌ **建议暂停**：存在根本性未验证风险，重新调研后再评估

**下一步行动**（3-5 条具体可执行）

## 偏见自查清单（贯穿全程）

| 偏见 | 表现 | 对策 |
|------|------|------|
| 证实偏见 | 只看支持自己的证据 | 主动寻找反对证据 |
| 乐观偏见 | 高估成功概率 | 先看数据再看叙述 |
| 投射偏见 | 以为用户需求和自己一样 | 真实访谈，不假设 |
| 权威偏见 | 大厂做了就认为该做 | 区分绝对与相对信息 |
| 叙述偏见 | 被好故事打动忽略数据 | 先看数字再读故事 |
| 规划谬误 | 低估时间成本风险 | 拆解流程逐阶段估算 |
| 损失反感 | 已投入资源不愿放弃 | 量化机会成本 |
| 社会认同偏见 | 跟风竞品 | 回归第一手用户数据 |
| 突出经验偏见 | 高估近期案例 | 关注基本比率 |
| 相对性偏见 | 过度依赖比较 | 建立绝对评估标准 |
| 喜好偏见 | 因喜欢某方向而美化数据 | 竞争性替代假设 |
| 稀缺偏见 | 觉得"错过就没了"仓促决策 | 情景分析评估时间窗口 |

## 信息不足时的引导问题

```
在开始分析之前，需要了解以下信息：

【必须了解】
1. 这个产品/方向要解决谁的什么问题？（目标用户 + 具体痛点）
2. 你们已经有哪些用户调研？访谈了多少人？
3. 竞品现在是怎么做的？

【有了更好】
4. 团队为什么觉得现在是做这件事的时机？
5. 内部有谁反对这个方向？他们的理由是什么？
6. 如果这个产品失败，你觉得最可能的原因是什么？
```

---

## 注意事项

- **不替用户做决策**：框架是辅助思考工具，判断权在产品团队
- **动态决策**：新信息进来后应更新判断，这是迭代过程
- **好问题优先**：比好流程更重要的是找到真正的关键问题
- **三的原则**：关键结论尽量来自 3 个独立信息源
- **数据要放入背景**：孤立数据无意义，需有参照系
- **写作即思考**：某层无法清晰表述，说明该层信息不足
